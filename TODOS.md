## P3: SkillKit collision protection (pre-scan)

**What:** Before calling `skillkit team install`, scan the target `skills_dir` for unmanaged paths (not in state.json's `owned_targets`) and warn the user about paths that SkillKit might clobber.

**Why:** `DeployResult.was_collision` is always `false` for SkillKitBackend ‚Äî SkillKit's stdout doesn't expose whether it overwrote unmanaged content. NativeBackend warns and skips; SkillKit silently clobbers. This is a regression in safety for users switching from native to SkillKit.

**Pros:** Restores the collision warning that NativeBackend provides. Prevents data loss for users who have manually-placed skills in the same directory.

**Cons:** ~30 lines of pre-scan logic in SkillKitBackend. Minor complexity addition to Phase 3.

**Context:** Identified during eng review of `docs/designs/swappable-skill-backends.md`. Pre-scan loads `state.json` owned_targets, lists `skills_dir` contents, and warns about any entry not in owned_targets before invoking skillkit. The scan is cheap (directory listing).

**Effort:** XS (human: ~1h / CC: ~5min)
**Priority:** P3
**Depends on / blocked by:** Phase 3 of swappable-skill-backends (SkillKitBackend)

**Why:** Lets users mix native (SHA-locked, cryptographically verified) for sensitive skills alongside SkillKit (marketplace, cross-agent) for community skills. Today the global backend applies to all skills.

**Context:** Deferred during CEO review of `docs/designs/swappable-skill-backends.md`. Add `backend: Option<String>` to `SkillDeclaration` struct, parsed from `skill.toml`. Factory resolves per-skill backend by falling back to global if unset.

## P2: `haven security-scan --fix`

**What:** When run with `--fix`, automatically remove source files that have High-severity findings, after a single confirmation prompt listing all affected files.

**Why:** The scan currently only reports findings. Users still have to manually `haven remove` each flagged file. `--fix` completes the loop.

**Pros:** Zero friction to clean up a repo once the scan runs. The confirmation prompt keeps it safe. High-severity-only scope avoids over-deleting on Medium/Low findings.

**Cons:** Destructive operation (deletes source files). Requires careful wording in the prompt.

**Context:** After `security_scan::run()` collects findings, filter to `Severity::High`, group by `src` path (multiple findings per file are common), prompt once ("Remove these N files? [y/N]"), then call `std::fs::remove_file` for each. The live file is left unchanged (same behavior as `haven remove`).

**Depends on / blocked by:** `haven security-scan` shipping.

## P2: `haven security-scan` pre-push hook

**What:** `haven init` (or a new `haven hooks install` command) writes a pre-push hook that runs `haven security-scan` and blocks the push if High-severity findings exist.

**Why:** The scan is only useful if it runs before secrets reach GitHub. A manual `haven security-scan` call is easy to forget; a pre-push hook makes it automatic.

**Pros:** Prevents the most common failure mode (accidental push). Works with both git and jj (jj supports `[ui] pre-push-hook`). Opt-in via `haven init --hooks`.

**Cons:** Adds a side effect to `init`. Hook management (install/uninstall) is new surface area.

**Context:** Git hook: `.git/hooks/pre-push` containing `#!/bin/sh\nhaven security-scan || exit 1`. Jj equivalent: `[ui]\npre-push-hook = "haven security-scan"` in `.jj/repo/config.toml`. Should respect `[security] scan_on_push = false` in `haven.toml` as an escape hatch.

**Depends on / blocked by:** `haven security-scan` shipping.

## P3: Skill default snippets convention (`haven-snippet*.md`)

**What:** Allow skills to ship a suggested default snippet that haven copies into
`ai/skills/<name>/all.md` (or platform-specific file) on install, instead of always

**Why:** Right now stubs are always blank. Once you control skills that have well-known
config patterns, shipping a default snippet removes friction for users installing those
skills ‚Äî they get a useful starting point without manual editing.

**Pros:** Zero user effort for a good starting point. Natural extension of the per-skill

**Cons:** Requires skill authors to adopt the convention; no value until you have
skills you control that can validate it.

**Context:** Deferred because no reliable external standard exists. Implementation: on

`haven-snippet.<platform-id>.md` in the skill root; if present, copy content into
the stub instead of leaving it empty. File naming uses `haven-snippet` prefix to
avoid collisions with skill's own content files.

**Depends on / blocked by:** ~~Managed config sections feature shipping~~ done; `haven ai add-local` shipping (that's the first place to validate the convention ‚Äî check for `haven-snippet.all.md` in the imported directory and use it as the `all.md` content instead of blank).

## P2: Configurable external diff tool for `haven diff`

**What:** Allow users to configure an external diff viewer (e.g. `delta`, `diff-so-fancy`) via `haven.toml` under a `[diff]` section: `tool = "delta"`.

**Why:** Some users have strong preferences for diff rendering. `haven diff` ships with built-in ANSI colors, but external tools like `delta` provide syntax highlighting, side-by-side views, and git-style decorations that can't be replicated in-process.

**Context:** `haven diff` landed with built-in `--color=auto|always|never` and `--stat`. External tool support is the next logical step once users start using the feature in practice. Implementation: pipe the full formatted diff string through `Command::new(tool).stdin(Stdio::piped()).spawn()` before printing. Only applies when the tool is configured and stdout is a tty.
