## Vision

### 10x Check

The 10x version is not a better dotfiles manager. It's **the package registry for AI developer environments** — what npm was to Node, but for AI tool configurations.

Today: developers Google "best Claude Code skills" and manually install things from GitHub. There's no standard format, no versioning, no dependencies, no discovery. `skills.sh` is the closest thing but it's interactive and ephemeral.

The 10x version: you describe your environment declaratively, haven resolves dependencies,
fetches versioned packages, installs everything. You publish your curated setup as a package
others can install in one command. A thriving ecosystem emerges around it.

```
haven bootstrap gh:popular-author/rust-ai-env
# Installs: 8 Claude Code skills, /commit + /review commands, MCP servers,
#           rustup + cargo-watch + bacon, mise: rust@stable node@lts,
#           dotfiles: .zshrc .gitconfig .cargo/config.toml
# Time: under 2 minutes on a fresh machine. Nothing else does this.
```

### Platonic Ideal

Declarative AI environment composition: you describe what kind of developer you are,
not what files you want. The system resolves the right combination.

```toml
[identity]
type = "developer"
focus = ["rust", "typescript", "devops"]
ai_assistant = "claude"
style = "minimal"

[ai]
skills = ["gh:gstack/standard-skills", "gh:rust-lang/claude-rust-skills"]
```

The user specifies *intent*. haven figures out what that requires.

### Delight Opportunities

1. `haven status` — colored, emoji-rich, grouped by module. Like `git status` but satisfying.
2. `haven bootstrap --dry-run` — full install plan preview before touching anything.
3. Auto-generated `~/.claude/CLAUDE.md` — Claude Code learns what's installed.
4. `haven publish` — share your environment as a versioned GitHub release.
5. Shell completions — `haven completions --shell zsh` for all commands + profile names.
6. `haven log` — changelog of what changed across machines and when.
7. Homebrew tap — `brew install haven-sh/tap/haven` closes the bootstrap paradox.

## Accepted Scope (added to this plan)

- `haven publish` — package config as versioned GitHub release (publishes to user's own repo, requires `gh auth`) — **NOT YET IMPLEMENTED**
- ✅ Auto-generated `~/.claude/CLAUDE.md` on every `haven apply` (reads SKILL.md YAML frontmatter; profile noted but not conditional) — **DONE (2026-03-20)** (`src/claude_md.rs`)
- Homebrew tap (`haven-sh/tap/haven`) + curl install script (`https://haven.sh/install`) — **NOT YET IMPLEMENTED**
  - Requires: CI/CD pipeline (GitHub Actions) producing pre-built binaries for macOS x86_64/aarch64 + Linux x86_64/aarch64
- Safe file watcher with sensitive-file detection: `.havenignore` + built-in sensitive-file blocklist
  - Blocklist: `*.env`, `*.pem`, `*_rsa`, `*_ed25519`, `.npmrc`, `.pypirc`, `*.pfx`, etc.
  - `.havenignore` checked FIRST (stop processing if matched); blocklist only runs on unignored files
  - Confirmation prompt applies to both `haven watch` events AND `haven add` commands
  - Skipped files logged to `~/.haven/skipped.log` (JSON: `{"file": "...", "reason": "...", "ts": "..."}`)
- `haven bootstrap --dry-run` — show full install plan without executing
- Shell completions: `haven completions --shell [zsh|bash|fish]` (via `clap_complete` crate)
- `haven log` — changelog of tracked changes with timestamps

## Deferred to TODOS.md

- Registry Protocol Standard: define `haven.pkg.toml` format when real usage data informs the design
- Cross-machine diff (`haven diff --machine <name>`): implement once 2+ machines have state.json files
- Conflict resolution UX: three-way diff when `apply` finds diverged files (source ≠ live ≠ last-known-good)

## Updated Architecture (post-expansion)

### New Commands (from accepted scope)

```
haven init               # initialize new haven repo
haven bootstrap          # full setup: apply + packages + ai (NEW: installs haven too via tap)
haven bootstrap gh:user/env  # bootstrap from a published environment package
haven apply              # apply all modules to current machine
haven apply --profile minimal
haven apply --module ai
haven sync               # detect and record changes (explicit mode)
haven watch              # start file watcher (opt-in, with safety net)
haven status             # show drift — colored, emoji-rich output
haven diff               # diff between source and live files
haven add ~/.zshrc       # explicitly add a new file to tracking
haven upgrade            # update all GitHub-sourced AI configs
haven publish            # NEW: package current config as versioned GitHub release
haven publish --tag v1.0 --name "my-ai-env"
haven completions --shell zsh  # NEW: shell completions
haven log                # NEW: changelog of changes across time
```

### Safety Net Architecture

```
File Watcher Event
      │
      ▼
.havenignore check ──── IGNORED ──── stop
      │
      ▼
Sensitive-file blocklist
      │
      ├── MATCH ──── Prompt user: "Block or track?"
      │                    │
      │              User confirms ──── track with explicit annotation
      │              User declines ──── skip, log to ~/.haven/skipped.log
      │
      └── CLEAN ──── auto-track
```

### CLAUDE.md Generation

After every `haven apply`, haven reads the installed skills and commands (from their
metadata files) and writes/updates `~/.claude/CLAUDE.md`:

```markdown
# Claude Code Environment
Generated by haven on {date} (profile: work)

## Installed Skills
- /office-hours: YC Office Hours partner — brainstorm and design
- /ship: Release engineer — merge, test, push, create PR

## Slash Commands
- /commit: Create conventional commit with good message

## MCP Servers
- memory: Neo4j graph memory (neo4j://localhost:7687)
- filesystem: Local file access (~/)
```

Haven reads skill metadata from: `{skill_dir}/SKILL.md` YAML frontmatter
(name, version, description) and command metadata from `{commands_dir}/*.md`.

### `haven publish` Architecture

```
haven publish --tag v1.0 --name "my-ai-env"

1. Check: `gh auth status` — fail loudly if not authenticated
2. Check: does current git remote repo exist and user has write access?
   If no repo: prompt "Create github.com/{user}/haven-config? [y/N]"
   If no write access: fail with clear message + suggestion
3. Read current haven.toml + all module configs
4. Create manifest: haven-manifest.json
   { "name": "my-ai-env", "version": "v1.0",
     "profiles": [...], "modules": [...],
     "author": "jstegeman", "created": "2026-03-20" }
5. Bundle: source files + module configs + manifest
   ALWAYS excludes: secrets.toml, any file matching blocklist
6. Create GitHub release via gh CLI:
   gh release create v1.0 --title "my-ai-env v1.0" bundle.tar.gz
7. Print: "Published! Others can bootstrap with:"
          "  haven bootstrap gh:jstegeman/haven-config"
```

**Source notation:** `gh:{owner}/{repo}` resolves to latest release of `github.com/{owner}/{repo}`.
`gh:{owner}/{repo}@{ref}` resolves to a specific tag/commit. The `@` syntax is consistent across
all source types (skills, commands, full env packages).

### Homebrew Tap Structure

```
github.com/haven-sh/homebrew-tap/
  Formula/
    haven.rb       # formula: download release binary for current arch
```

Install script (`https://haven.sh/install`):

```bash
#!/bin/sh

# 2. Download pre-built binary from GitHub releases
# 3. Install to /usr/local/bin/haven (or ~/.local/bin on Linux)

```

## Open Questions (carry forward from design doc)

1. **DECIDED (2026-03-20):** File naming — **chezmoi-compatible magic-name encoding**
   chosen over "plain names + TOML dest mapping". Source filenames like `dot_zshrc`,
   `private_dot_ssh/id_rsa`, `executable_dot_local/bin/foo` encode all metadata with
   no TOML registry. Enables zero-friction migration from chezmoi and compatible tooling.
2. **DECIDED:** Lockfile format: `dfiles.lock` pins GitHub sources by commit SHA (implemented).
3. **DECIDED:** Conflict resolution — default is backup-and-overwrite for MVP:
   `dfiles apply` backs up the live file to `~/.dfiles/backups/` before overwriting.
   Three-way diff tool deferred to post-v1 (TODOS.md).
4. **DECIDED:** Rust chosen for implementation.
5. **OPEN:** `dfiles publish` bundle format — tar.gz vs directory — and whether secrets.toml
   is excluded by default or opt-in exclude.

## Temporal Interrogation

Implementation decisions to resolve before coding:

**HOUR 1 (foundations):**
- Which Rust crates? Recommend: `clap` (CLI), `tera` (templates), `toml` (config), `notify` (file watch), `reqwest` (HTTP for GitHub), `serde` (serialization)
- File structure: does `dfiles init` live inside a git repo (like chezmoi) or standalone directory?

**HOUR 2-3 (core logic):**
- How does apply handle files that already exist at the destination? Options: overwrite, backup, ask, fail.
- How does the symlink vs copy decision work? chezmoi copies; GNU Stow symlinks. Each has tradeoffs.

**HOUR 4-5 (integration):**
- 1Password subprocess: synchronous `op read` or batch? Batch is faster but requires waiting for all templates to be scanned first.
- GitHub rate limiting: unauthenticated API is 60 req/hour. For users with many gh: sources, this hits fast. Need `gh auth token` fallback.

**HOUR 6+ (polish/tests):**
- Shell completions: `clap_complete` crate handles this cleanly; wire it early.
- First-run experience: does `dfiles bootstrap` on a machine with no config ask questions or fail loudly?

## Feasibility Notes

- **1Password is optional.** If `op` CLI not installed, secrets module is skipped with a warning.
  Fallback: plain-text templates can use `{{ env "VAR_NAME" }}` as an alternative.
  No hard dependency on 1Password for core functionality.

- **GitHub API rate limiting.** Unauthenticated: 60 req/hour. Mitigation:
  (1) cache fetched GitHub sources in `~/.dfiles/cache/` with TTL;
  (2) prefer `gh auth token` when available;
  (3) batch all GitHub fetches at bootstrap start rather than inline.

- **Homebrew tap requires CI/CD.** GitHub Actions workflow to build and publish
  pre-built binaries for: macOS x86_64 (Intel), macOS aarch64 (Apple Silicon),
  Linux x86_64, Linux aarch64. This is boilerplate (`cross` crate + release workflow)
  but is ongoing maintenance. Estimate: 1 day to set up (human) / 15 min (CC).

- **SKILL.md YAML frontmatter format** (for CLAUDE.md generation):
  ```yaml
  ---
  name: office-hours
  version: 1.0.0
  description: "YC Office Hours partner — brainstorm and design"
  ---
  ```
  Fields `name` and `description` are required for CLAUDE.md generation.
  If missing or malformed, skill is listed with its directory name and "(no description)".
