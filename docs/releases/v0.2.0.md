# dfiles v0.2.0 Release Notes

**Release date:** 2026-03-21

---

## Highlights

### `dfiles ai add-local` — import your own skills

```sh
dfiles ai add-local ~/.claude/skills/my-skill
dfiles ai add-local ~/dev/my-skill --name myskill --platforms claude-code
```

Moves a locally developed skill directory into the dfiles repo under
`ai/skills/<name>/files/`, writes a `skill.toml` with `source = "repo:"`,
creates a blank snippet stub, and removes the original. Run `dfiles apply --ai`
afterward to deploy the symlink.

This is the bridge between developing a skill locally and managing it with
dfiles. The `repo:` source type means dfiles reads the skill directly from
your repo — no download, no cache, no lock entry.

---

### `extfile_` — track single-file and archive remote downloads

```toml
# source/extfile_dot_local_bin_delta
type = "file"
url  = "https://github.com/dandavison/delta/releases/download/0.18.2/delta-0.18.2-aarch64-apple-darwin.tar.gz"
sha256 = "abc123..."   # optional but recommended
```

Two types:
- `type = "file"` — download the URL directly to the destination
- `type = "archive"` — download a tarball and extract a specific `subpath` from it

`dfiles apply` downloads on the first run, verifies SHA-256 if provided.
`dfiles diff` shows `?` when the destination is absent.

### Managed config section injection

`dfiles apply --ai` now injects skill snippets into platform config files (e.g.
`~/.claude/CLAUDE.md`) between comment markers:

```
<!-- dfiles managed start -->
...snippet content...
<!-- dfiles managed end -->
```

The markers are added once (with a prompt if the file already exists and has no
markers). All subsequent applies update the content in-place. Works for both
`all.md` (every platform) and `<platform-id>.md` (platform-specific).

---

### `dfiles ai search` — discover skills from the registry

```sh
dfiles ai search jujutsu
dfiles ai search "pdf processing" --limit 5
```

Searches the [skills.sh](https://skills.sh) registry. Results show the skill
source in `gh:owner/repo/skill` format and install count. Copy the source and
pass it to `dfiles ai add` to start tracking.

---

### `dfiles ai scan` — import existing unmanaged skills

```sh
dfiles ai scan ~/.claude/skills
dfiles ai scan ~/.claude/skills --dry-run
```

Walks the given directory for skill subdirectories. For each unmanaged skill:
1. Tries to detect the GitHub source via git remote
2. Falls back to skills.sh fuzzy search on the skill directory name
3. Prompts you to confirm, edit the source, or skip

Skills already in `ai/skills.toml` are silently skipped.

---

### `dfiles diff` improvements

- **AI skill SHA drift** — shows `~ name  (installed: abc12345, pinned: def67890)`
  when a deployed skill is behind its lock entry.
- **`extdir_` ref drift** — shows `M path  (extdir: at abc12345, expected v3.0.0)`
  when a cloned external repo is behind the pinned ref.

---

### Supply chain protection

`dfiles.lock` SHA verification is now enforced on every cache miss. If the
freshly-fetched content SHA doesn't match the lock entry, `dfiles ai fetch`
and `dfiles apply --ai` hard-fail with a message pointing to
`dfiles ai update`. This prevents silent substitution attacks on `gh:` sources.

---

### Performance and reliability

- **Parallel skill fetches** — multiple `gh:` cache misses are fetched
  concurrently using `std::thread::scope`. No new dependencies.
- **Apply file lock** — PID-based lock at `~/.dfiles/apply.lock` prevents
  concurrent apply runs from corrupting state. Stale locks are auto-cleaned.

---

### `dfiles source-path` and XDG default

```sh
cd $(dfiles source-path)
```

New `source-path` command prints the resolved repo path. New installs default
to `~/.local/share/dfiles` (XDG standard). Existing `~/dfiles` repos are
auto-detected without any migration step.

---

## Other changes

- `dfiles add --update` — re-copy an already-tracked file without an error
- `home_dir` template variable in `.tmpl` files
- `create_` prefix support — files are written only if the destination doesn't exist
- `exact_` directory prefix — untracked destination files are removed on apply
- Script execution — `dfiles apply --run-scripts` runs `source/scripts/`; `run_once_` tracked in state
- Opt-in local telemetry to `~/.dfiles/telemetry.jsonl`
- `modify_` scripts in chezmoi import now emit a clear skip message

---

## Upgrade notes

No breaking changes from v0.1.0.

The XDG default path (`~/.local/share/dfiles`) only applies to new installs.
If you have an existing repo at `~/dfiles`, dfiles will continue to use it
automatically.
