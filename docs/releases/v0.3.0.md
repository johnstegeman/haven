# dfiles v0.3.0 Release Notes

**Release date:** 2026-03-23

---

## Highlights

### Security fix — please upgrade from v0.2.0

A HIGH-severity vulnerability was found and patched in this release.
**If you use `gh:` skill sources, upgrade before running `dfiles apply`.**

A malicious tarball served by a compromised `gh:` skill could plant a symlink
entry pointing outside the skill cache directory. A subsequent extraction step
would follow the symlink and write files to an attacker-controlled path — on a
developer machine, this means shell startup files can be overwritten (RCE via
next shell open). The fix unconditionally skips symlink and hard-link tar
entries before extraction. Two regression tests were added.

See [`docs/designs/`](../designs/) for the full audit record.

### `dfiles security-scan` — audit your tracked files for secrets

```sh
dfiles security-scan          # scan all tracked files
dfiles security-scan --entropy  # also flag high-entropy strings
```

Checks every file in `source/` for:

| Check | What's flagged |
|-------|---------------|
| Filename | `.env`, `id_rsa`, `credentials`, `.pem`, `.key`, `.p12`, etc. |
| Path | `~/.aws/credentials`, `~/.kube/**`, `~/.ssh/**`, `~/.config/gh/hosts.yml`, `~/.docker/config.json`, etc. |
| Content | GitHub tokens (`ghp_`/`ghs_`/`github_pat_`), AWS access keys (`AKIA…`), PEM private keys, OpenAI keys (`sk-…`), Anthropic keys (`sk-ant-…`), generic `password =`/`secret =` assignments |
| High-entropy (opt-in) | Random-looking tokens ≥16 chars, Shannon entropy >4.5 bits/char |

Exits 0 when clean, 1 when findings are reported. CI-friendly.

**False positives:** suppress with `[security] allow` in `dfiles.toml`:

```toml
[security]
allow = [
  "~/.config/gh/hosts.yml",
  "~/.config/gcloud/**",
]
```

**Integrated into `dfiles add`:** content is scanned automatically when you
add a file. If secrets are found, you are prompted before the file is saved to
`source/`. Declining removes it immediately.

### Shell completions

```sh
# Fish — write once:
dfiles completions fish > ~/.config/fish/completions/dfiles.fish

# Zsh — add to ~/.zshrc:
source <(dfiles completions zsh)

# Bash — add to ~/.bashrc:
source <(dfiles completions bash)
```

All subcommands and flags are included, including the new `security-scan` and
`completions` commands themselves.

### Jujutsu (jj) VCS backend

```sh
# Set globally in dfiles.toml:
[vcs]
backend = "jj"

# Or per-command:
dfiles init gh:you/env --vcs jj

# Or via env var:
DFILES_VCS=jj dfiles init
```

When jj is configured, all new `extdir_` clones and `dfiles init` operations
use `jj git clone --colocate` instead of `git clone`. Repos get both `.jj/`
and `.git/`, so both tools work in them.

On first use without a config, dfiles detects whether jj is on your PATH and
prompts once (save, use-once, or stay with git). The choice is persisted in
`dfiles.toml`.

```sh
dfiles vcs   # show active backend and how it was resolved
```

## Other changes

- **`config/ignore` warning on `dfiles add`** — adding a file that matches an
  ignore pattern now shows a clear message instead of silently succeeding.
- **Installer guide section** — the installation documentation was expanded
  with step-by-step instructions.

## Upgrade notes

No breaking changes. Existing repos work without modification.

If you track `~/.ssh/**`, `~/.kube/**`, or similar paths, `dfiles security-scan`
will flag them as expected. Add them to `[security] allow` in `dfiles.toml` to
suppress the findings if you're intentionally tracking those files.
