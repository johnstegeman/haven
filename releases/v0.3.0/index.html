haven v0.3.0 Release Notes¶

Release date: 2026-03-23

Highlights¶

Security fix — please upgrade from v0.2.0¶

A HIGH-severity vulnerability was found and patched in this release. If you use gh: skill sources, upgrade before running haven apply.

A malicious tarball served by a compromised gh: skill could plant a symlink entry pointing outside the skill cache directory. A subsequent extraction step would follow the symlink and write files to an attacker-controlled path — on a developer machine, this means shell startup files can be overwritten (RCE via next shell open). The fix unconditionally skips symlink and hard-link tar entries before extraction. Two regression tests were added.

See docs/designs/ for the full audit record.

haven security-scan — audit your tracked files for secrets¶

haven security-scan          # scan all tracked files
haven security-scan --entropy  # also flag high-entropy strings

Checks every file in source/ for:

Check What's flagged
Filename .env, id_rsa, credentials, .pem, .key, .p12, etc.
Path ~/.aws/credentials, ~/.kube/**, ~/.ssh/**, ~/.config/gh/hosts.yml, ~/.docker/config.json, etc.
Content GitHub tokens (ghp_/ghs_/github_pat_), AWS access keys (AKIA…), PEM private keys, OpenAI keys (sk-…), Anthropic keys (sk-ant-…), generic password =/secret = assignments
High-entropy (opt-in) Random-looking tokens ≥16 chars, Shannon entropy >4.5 bits/char

Exits 0 when clean, 1 when findings are reported. CI-friendly.

False positives: suppress with [security] allow in haven.toml:

[security]
allow = [
  "~/.config/gh/hosts.yml",
  "~/.config/gcloud/**",
]

Integrated into haven add: content is scanned automatically when you add a file. If secrets are found, you are prompted before the file is saved to source/. Declining removes it immediately.

haven upgrade — self-update in one command¶

haven upgrade           # upgrade to the latest version
haven upgrade --check   # check without installing (exits 1 if update available)
haven upgrade --force   # reinstall the current version

haven now upgrades itself. It fetches the latest release from GitHub, verifies the SHA256 checksum, and atomically replaces the running binary via rename. No need to re-run the install script.

The --check flag is CI-friendly — it exits 0 when up to date and 1 when an update is available, making it easy to add upgrade checks to setup scripts.

[data] custom template variables¶

Define your own variables in haven.toml and use them in any .tmpl file:

[data]
host = "my-laptop"
kanata_path = "/usr/local/bin/kanata"

# In any .tmpl file:
export HOST={{ data.host }}
{{ if data.host }}# host-specific config{% endif %}

Variables are accessible as {{ data.<key> }}. Nested objects and non-string values are not supported — only flat string keys.

haven data prints all variables currently in scope:

haven data

os         = macos
hostname   = my-laptop
username   = alice
home_dir   = /Users/alice
source_dir = /Users/alice/.local/share/haven

data.host                = my-laptop
data.kanata_path         = /usr/local/bin/kanata

Chezmoi import: haven import --from chezmoi automatically migrates .chezmoidata.yaml and .chezmoidata.toml into [data] entries in haven.toml. The template converter also updates .tmpl files — chezmoi's {{ .host }} becomes {{ data.host }} and {{ if (index . "host") }} becomes {% if data.host %}.

haven unmanaged — discover dotfiles you haven't added yet¶

haven unmanaged                     # scan ~ for untracked dotfiles
haven unmanaged --path ~/.config    # scan a specific directory
haven unmanaged --depth 5           # recurse deeper (default: 3)

Walks your home directory and reports every file that isn't tracked by haven. At the home root, only dotfiles and dotdirs (names starting with .) are examined — Documents/, Downloads/, Projects/, etc. are always skipped. High-noise directories are skipped at every depth:

Category Directories skipped
VCS internals .git, .jj, .hg, .svn
Build / caches node_modules, target, .cache, .cargo, .npm, .rustup, .pyenv, .rbenv, .nvm, .volta, .asdf, .mise, .gradle, .m2
macOS system Library, .Trash, .Spotlight-V100, .DocumentRevisions-V100, .fseventsd, .TemporaryItems
App-managed .android, .minikube, .kube, .docker, .vscode-server, .cocoapods, .bundle
haven state .haven

Pipe the output to haven add to bulk-import discovered files:

haven unmanaged | xargs -I{} haven add {}

haven list — see what you're tracking¶

haven list

~/.zshrc
~/.gitconfig          (template)
~/.ssh/config         (private)
~/.local/bin/delta    (extfile)
~/.config/nvim        (extdir)

One line per tracked file, decoded from source/ magic names into the destination path you actually care about. Flag annotations show when something unusual is happening to the file — private means 0600, template means it's rendered through Tera before writing, extdir means it's a remote git clone, and so on.

Files matching config/ignore patterns are excluded, same as apply and status.

haven telemetry — manage telemetry from the CLI¶

haven telemetry --enable   # set [telemetry] enabled = true in haven.toml
haven telemetry --disable  # set [telemetry] enabled = false in haven.toml
haven telemetry            # show current status
haven telemetry --note "starting fresh config — prior data is from testing"

Previously, enabling telemetry required manually editing haven.toml. --enable and --disable handle it with one command (using toml_edit internally, so your existing comments and formatting are preserved).

--note appends a free-form annotation to ~/.haven/telemetry.jsonl regardless of whether telemetry is enabled. Use it to mark test runs, onboarding sessions, or observed issues — the notes become context when analysing the telemetry log later.

Shell completions¶

# Fish — write once:
haven completions fish > ~/.config/fish/completions/haven.fish

# Zsh — add to ~/.zshrc:
source <(haven completions zsh)

# Bash — add to ~/.bashrc:
source <(haven completions bash)

All subcommands and flags are included, including the new security-scan and completions commands themselves.

Jujutsu (jj) VCS backend¶

# Set globally in haven.toml:
[vcs]
backend = "jj"

# Or per-command:
haven init gh:you/env --vcs jj

# Or via env var:
HAVEN_VCS=jj haven init

When jj is configured, all new extdir_ clones and haven init operations use jj git clone --colocate instead of git clone. Repos get both .jj/ and .git/, so both tools work in them.

On first use without a config, haven detects whether jj is on your PATH and prompts once (save, use-once, or stay with git). The choice is persisted in haven.toml.

haven vcs   # show active backend and how it was resolved

Other changes¶

  • Getting started guide for chezmoi users — new docs/from-chezmoi.md walks existing chezmoi users through migration. Covers the automated import path, template syntax conversion, command equivalence table, Brewfile/module/ profile setup, AI skill import, new-machine bootstrap, and a gap table of chezmoi features not yet in haven.
  • config/ignore warning on haven add — adding a file that matches an ignore pattern now shows a clear message instead of silently succeeding.
  • Installer guide section — the installation documentation was expanded with step-by-step instructions.

Upgrade notes¶

No breaking changes. Existing repos work without modification.

If you track ~/.ssh/**, ~/.kube/**, or similar paths, haven security-scan will flag them as expected. Add them to [security] allow in haven.toml to suppress the findings if you're intentionally tracking those files.

What's next (planned for v0.4.0)¶

  • Pre-push hook support (haven init --hooks)
  • --from yadm and --from mackup importers
  • brew install haven distribution via Homebrew tap
  • Periodic automatic update checks ([update] check = "weekly" in haven.toml)
