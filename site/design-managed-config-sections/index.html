Design: Managed Config Sections for AI Skills¶

Overview¶

Skills can ship per-platform config snippets. The user stores their agent config file (e.g. ~/.claude/CLAUDE.md) in source/ with HTML comment markers. On dfiles apply, dfiles replaces the content between markers with assembled snippets from all installed skills for that platform.

Skill Snippet Files¶

Snippets live in the user's dfiles repo under ai/skills/<name>/, co-located with the skill declaration:

ai/
  skills/
    pdf-processing/
      skill.toml        ← source, platforms, deploy (replaces ai/skills.toml entries)
      all.md            ← injected into every platform's config_file (blank by default)
      claude-code.md    ← platform-specific snippet (optional)
  platforms.toml

When a skill is added (dfiles ai add / dfiles ai scan), a blank all.md stub is created automatically and the user is notified:

✓ pdf-processing  cached (abc1234)
  → snippet: ai/skills/pdf-processing/all.md (edit to add agent instructions)

The user edits it to add their own instructions.

Skills do not ship default snippets — stubs are always blank. A future convention (dfiles-snippet*.md in the fetched skill root) may allow opt-in defaults once there are skills that can validate the pattern.

When assembling the managed section for a platform, order is: all.md first, then <platform-id>.md, in skill declaration order. Missing files are silently skipped.

Structure migration: ai/skills.toml (flat [[skill]] array) is replaced by per-directory ai/skills/<name>/skill.toml files. Since dfiles is pre-release, this is a clean break with no backwards compatibility shim needed.

Marker Format¶

HTML comments — agents see them in raw markdown but skip them when reasoning about content:

<!-- dfiles managed start -->
<!-- dfiles managed end -->

Source File (dfiles repo)¶

User creates source/dot_claude/CLAUDE.md with their own content and empty markers wherever they want the managed section:

# My Claude Setup

Some user-managed content...

<!-- dfiles managed start -->
<!-- dfiles managed end -->

More user content below...

The source file always has empty markers. Generated content never lives in source/.

Apply Behavior¶

After files are copied and skills are deployed, a config injection step runs for each platform with a config_file defined:

  1. Destination file exists and contains markers → inject snippets between them
  2. Destination file exists, no markers, snippets exist →
  3. TTY: offer interactive prompt to add markers to source/ file (beginning/end/skip)
  4. No TTY: print a one-line notice, skip
  5. Destination file does not exist, no source file in repo, snippets exist → create a stub with a minimal header + empty markers, then inject
  6. Destination file does not exist, source file exists in repo → skip stub creation; files phase is responsible for the copy (or user needs to run apply with files phase)
  7. Destination file does not exist, no snippets → do nothing (no auto-generation)

ApplyOptions gets an interactive: bool field (derived from atty::isatty(stdin) in production; set explicitly in tests).

Managed Section Content¶

The managed section contains two kinds of generated content, in order:

  1. Installed skills/commands list — the registry of what dfiles has deployed (currently auto-generated by claude_md.rs)
  2. Skill snippets — all.md + <platform-id>.md from each skill's directory, in declaration order

The user may also maintain their own skills list outside the markers (curated/custom). Agents reading both lists will understand one is declarative (user-authored) and one is a live dfiles registry. No conflict expected.

Example managed section output:

<!-- dfiles managed start -->
<!-- managed by dfiles (dfiles apply to regenerate) -->

## Installed Skills
- /pdf-processing: Process and analyze PDF files
- /find-skills: Discover available AI skills

## Slash Commands
- /commit: Create conventional commits

<!-- skill: pdf-processing -->
## PDF Processing
When working with PDFs, use the /pdf skill...
<!-- /skill: pdf-processing -->

<!-- skill: find-skills -->
...
<!-- /skill: find-skills -->

<!-- dfiles managed end -->

CLAUDE.md Auto-generation¶

claude_md.rs currently generates the entire ~/.claude/CLAUDE.md. New behavior:

  • Never generate the file from scratch, except case 3 above (stub with minimal header + empty markers only)
  • Existing skills/commands list generation moves into managed section injection
  • claude_md.rs is repurposed to generate only the managed section content; rename or consolidate into a new config_injection.rs

Adding Markers Interactively¶

When apply encounters a config file without markers and skills have snippets for that platform (TTY only):

dfiles: ~/.claude/CLAUDE.md has no managed section.
Skills have snippets to inject — where should the section go?
  [B] beginning of file
  [E] end of file
  [S] skip (don't ask again for this file)

Non-interactive (no TTY): print a notice instead of prompting:

note: 3 skill snippets pending for ~/.claude/CLAUDE.md (no managed section) — run dfiles apply in a terminal to configure

[S] skip persistence: The skipped file path is recorded in state.json under skipped_managed_files: Vec<String> so the prompt doesn't re-appear on every apply.

If yes: insert the empty markers at the chosen position in the source/ file (not the live copy). Live copy updates during the current apply run.

If no source file exists yet: offer to import the live file into source/ first (stripping any injected content between markers before saving).

Importing Files with Markers (dfiles add)¶

When a user adds a file to dfiles that contains markers, strip everything between them before saving to source/. This keeps the repo clean.

dfiles diff should warn if markers in a source file are non-empty (user accidentally committed generated content).

CLI Output¶

On injection (apply): Print only when something changed (silent if idempotent):

✓ ~/.claude/CLAUDE.md  (managed section updated: 3 skills)

Dry run (--dry-run): Full file diff in unified diff format, consistent with dfiles diff output style:

~ ~/.claude/CLAUDE.md  (managed section)
  --- current
  +++ after apply
  @@ -8,2 +8,12 @@
   <!-- dfiles managed start -->
  +<!-- managed by dfiles (dfiles apply to regenerate) -->
  +
  +## Installed Skills
  +- /pdf-processing: Process and analyze PDF files
  +
  +<!-- skill: pdf-processing -->
  +When working with PDFs...
  +<!-- /skill: pdf-processing -->
  +
   <!-- dfiles managed end -->

For newly created stubs: show the full file as an addition (+++ new file).

Key Implementation Touchpoints¶

File Change
src/config_injection.rs New: marker detection, snippet assembly, file patching, stub creation
src/ai_skill.rs Add snippet_path(platform_id) -> Option<PathBuf> for a deployed skill
src/commands/apply.rs Call inject_config_snippets() after apply_ai_skills(); wire dry-run diff output
src/claude_md.rs Repurpose to generate managed section content only; no more full-file generation
src/commands/add.rs Strip managed section content on import if markers detected
src/commands/diff.rs Warn if source file has non-empty markers; show injection diff
src/data/platforms.toml No changes; config_file field already identifies the target

Edge Cases¶

  • Symlinked config file: injecting into a symlinked destination would modify the source file in the repo. Warn and skip; document that this feature requires copy deployment for config files with managed sections.
  • Mismatched markers: if only start or only end is present, emit a clear warning and skip injection — do not produce corrupted output silently.
  • skill.toml parse error: warn and skip that skill; do not abort apply.
  • Markers missing from destination: offer interactively (TTY) or notice (no TTY).
  • Skill has no snippet files: skip silently.
  • Re-apply: full replace of managed section ensures idempotency.
  • Stub creation: only if no source file in repo AND at least one skill has a snippet file for that platform.
  • Stub content: minimal — platform name as H1 heading + empty markers only: ```markdown # Claude Code

`` - **ai/skills/absent**:fs::create_dir_all` before stub write.
